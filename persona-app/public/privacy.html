Personal Layer

Effective date: June 23, 2026

Privacy Policy

Personal Layer is a local-first personal context system. It helps approved apps and AI assistants understand your preferences by building a private profile on your device. This policy explains what data is processed, how Google user data is handled, what stays local, what limited metadata may be stored in the cloud, how long data is retained, and how to contact us.

Google User Data Accessed

If you connect Google integrations, Personal Layer requests only the Google scopes needed for the features you choose: Gmail read-only access, Calendar read-only access, YouTube read-only access, and Google Drive metadata read-only access. Current Google OAuth scopes may include:

  • https://www.googleapis.com/auth/gmail.readonly
  • https://www.googleapis.com/auth/calendar.readonly
  • https://www.googleapis.com/auth/youtube.readonly
  • https://www.googleapis.com/auth/drive.metadata.readonly

For Gmail, Personal Layer uses message metadata such as message IDs, thread IDs, labels, snippets, sender domains, timestamps, and history IDs. For Calendar, it uses event metadata such as event IDs, summaries, start and end times, status, created and updated times, duration, and attendee counts. For Drive, it uses file metadata such as file IDs, names, MIME types, folder or parent information, ownership metadata, and edit timing. For YouTube, it uses read-only viewing and activity metadata available through the connected account.

How Google User Data Is Used

Google user data is used to build user-controlled personal context such as work patterns, availability patterns, content category preferences, workflow hints, and integration sync status. The product uses these derived signals to help the user manage their personal context and to provide scoped context bundles to apps only after user authorization.

Personal Layer does not use Google user data for advertising, does not sell Google user data, and does not use Google Workspace API data to develop, improve, or train generalized AI or machine learning models.

Google User Data Sharing

Personal Layer does not share raw Google user data with third parties. Apps connected through Personal Layer may receive only user-approved, scoped, filtered context bundles, such as abstract preference signals, availability labels, feature usage patterns, provenance metadata, and confidence values.

Service providers used to operate hosting, storage, security, diagnostics, and delivery may process limited operational metadata only as needed to provide the service. We do not permit service providers to use Google user data for their own advertising, profiling, or model training.

Other Data We Process

Depending on what you enable, Personal Layer may process app and browser interaction events, connector metadata from services such as email, calendar, GitHub, Notion, Spotify, YouTube, or Google services, derived profile signals, permission records, privacy boundaries, audit logs, device-pairing records, sync status, and account metadata needed to operate cloud-backed services.

Personal Layer is designed to filter or block secrets, credentials, payment data, precise private content, and sensitive personal identifiers before data is shared with apps.

What Stays Local

Raw event payloads, raw email bodies, private notes, browser page text, local files, synthesized behavioral profiles, local memory graphs, embedding vectors, local query logs, domain approvals, privacy boundaries, local vault data, and OAuth tokens stay on your device unless you explicitly export them, sync them to a trusted paired device, or authorize sharing.

Local data is encrypted at rest where supported by the runtime. Production builds should use operating-system keychain or secure storage for device keys and tokens.

Cloud Storage

The production cloud service is intentionally thin. It may store account identity metadata, developer registry records, API key hashes and prefixes, app permission grants, device and pairing metadata, encrypted summary blobs, push tokens and routes, and redacted operational telemetry.

We do not sell personal data, train foundation models on your personal context, or store raw persona events in the cloud.

How Apps Receive Context

Apps can receive context only after consent. A context bundle may include abstract preference signals, active context labels, coarse feature usage patterns, capability or workflow hints, confidence, and provenance metadata.

Apps must not receive raw connector payloads, specific email bodies, passwords, access tokens, payment card numbers, social security numbers, exact private locations, or raw browsing histories through standard Personal Layer APIs.

Your Controls

You can approve or deny app access by scope, revoke access, disconnect integrations, create privacy boundaries, delete supported signals, delete local context data, and request account deletion and deletion of cloud metadata.

Revocation stops future access. Apps and developers that received context before revocation are responsible for honoring applicable developer terms and deletion requirements.

Retention And Security

Raw local events are retained for 90 days by default unless you choose a shorter retention period. Local query logs are retained for 30 days by default. Cloud permission and account metadata is retained while the account is active, and operational telemetry is retained for up to 30 days unless needed for security investigation or legal compliance. OAuth tokens and sync metadata are retained only while the integration remains connected.

Security controls include local encryption, OS keychain storage for tokens where supported, encrypted token storage, scoped app permissions, row-level security on cloud tables, CORS and origin checks, filtered outbound bundles, audit logs, and silent push payloads without behavioral text.

Data Deletion

You can disconnect Google integrations in the app to stop future access and remove stored OAuth tokens. You can delete supported local signals and request account deletion and deletion of cloud metadata.

To request deletion of Google user data, cloud metadata, or your account, contact privacy@personallayer.dev. We will delete or de-identify data unless retention is required for security, legal, or abuse-prevention reasons.
